Принципы Работы Агента

Если пользователь явно не попросил иначе, агент должен:

    предпочитать минимальное безопасное изменение, которое решает задачу;
    сохранять существующую архитектуру и naming conventions;
    проверять актуальное состояние репозитория перед нетривиальными изменениями;
    обновлять тесты, docs и примеры, если они устаревают из-за изменений;
    проверять результат перед завершением работы;
    не делать спекулятивных рефакторингов;
    спрашивать разрешение перед каждой правкой кода;
    спрашивать перед разрушительными, необратимыми, дорогими или влияющими на production действиями.

    Жёсткие ограничения: не трогай без разрешения файлы .env и src/config.py

Стратегия Проверки

Обычно агент должен валидировать изменения в таком порядке:

    file-level проверка измененных файлов;
    синтаксическая проверка Python без побочных эффектов;
    ближайшие сценарии FSM / callback flow, которые были затронуты;
    full-repo проверки при необходимости.

Если compileall не работает из-за ограничений записи в __pycache__, допустима проверка через compile(...) в памяти.

Стиль Кода И Naming

    Formatter: форматировать табами, если рядом код уже ведется в таком стиле
    Linter: ruff
    Политика типизации: Static
    Политика комментариев: без лишних комментариев, разрешены только короткие docstring функций
    Политика импортов: grouped
    Подход к обработке ошибок: exceptions + понятное сообщение пользователю
    Подход к конфигурации: central config в src/config.py
    FSMContext называй state

Делать:

    использовать названия, отражающие намерение;
    держать модули цельными и сфокусированными;
    следовать уже существующим паттернам рядом с изменяемым кодом;
    соблюдать pep-8;
    предпочитать явную бизнес-логику “умным” абстракциям;
    в callback/FSM логике явно фиксировать, какие данные лежат в state.

Не делать:

    превращать utils в свалку несвязной логики;
    смешивать несколько naming styles в одной зоне кода;
    прятать важные side effects за расплывчатыми helper names;
    вводить широкие абстракции до появления второго реального use case;
    дублировать генерацию ИИ для одного и того же проекта без необходимости.

Правила

    Не редактировать generated files вручную, если в проекте позже появится regeneration workflow.
    Сохранять backward compatibility для callback_data, state-ключей и DB слоя, если задача явно не разрешает breaking change.
    При изменении пользовательского flow обновлять связанные клавиатуры, состояния и обработчики вместе.
    При изменении контракта prompt -> parser обновлять обе стороны одновременно.

Границы Безопасности И Safety

Считайте этот раздел обязательным.

Жёсткие Правила

    Никогда не коммитить secrets, private keys, access tokens или production credentials.
    Никогда не хардкодить secrets в source code, tests, fixtures или documentation.
    Редактировать чувствительные значения в логах и примерах.
    Валидировать и санитизировать недоверенный ввод на границе Telegram -> bot.
    Быть особенно аккуратным в коде, связанном с логинами и паролями Kwork, .env и src/config.py.

Перед Этими Действиями Нужна Подтверждённая Проверка Человеком

    удаление данных или файлов;
    изменение auth или permission logic;
    изменение схемы хранения логинов, паролей или токенов;
    изменение .env;
    изменение src/config.py;
    установка или замена крупных зависимостей;
    ротация secrets или изменение security configuration.

Чувствительные Зоны

    src/config.py
    .env

Definition Of Done

Изменение не считается завершённым, пока:

    не прошли релевантные проверки;
    не обновлены docs/config/examples, если они затронуты;
    размещение файлов и naming соответствуют этому документу;
    assumptions, risks и follow-up work зафиксированы;
    пользователь был спрошен перед каждой правкой кода.

Когда Агент Должен Остановиться И Спросить

Агент должен остановиться и спросить человека, если:

    требования неоднозначны и существует несколько валидных реализаций;
    изменение может нарушить совместимость callback flow, state keys или безопасность хранения данных;
    документация и код существенно противоречат друг другу;
    тесты падают по причинам, не связанным с задачей, и причина неясна;
    для задачи нужны secrets, production access или product-policy decisions;
    самый безопасный путь зависит от tradeoff, который пользователь ещё не выбрал;
    требуется правка .env или src/config.py.
